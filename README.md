# Шаблон проекта ШИФТ Интенсив

Шаблон, на основе которого можно начинать проект

## Запуск

1. Запускаем Apache Cassandra (docker-compose.yaml)
2. Накатываем скрипт scripts/init.cql на БД
3. Запускаем сервис

## Технологии/зависимости

### Java

В рамках проекта используется Java версией 17. Скачать можно по ссылке https://bell-sw.com/pages/downloads/#downloads

### Среда разработки

Мы используем Jetbrains IDEA и вам советуем. Доступна тут https://www.jetbrains.com/idea/download/

Community Edition будет достаточно

### Gradle

Сборка проекта осуществляется с помощью Gradle версией 7. Для получения Gradle достаточно из проекта вызвать команду

```console
gradlew
```

### База данных

В качестве БД используем Apache Cassandra 4. Скачать и запустить можно с помощью Docker команды из проекта

```console
docker-compose up -d
```

### Структура проекта

1. Класс ru.cft.shift.intensive.template.Application - точка входа в приложение. Запуск приложения будет осуществляться
   с помощью метода main
2. Пакет ru.cft.shift.intensive.template.configuration - здесь хранятся все конфигурации
3. Пакет ru.cft.shift.intensive.template.controller - пакет с классами API
4. Пакет ru.cft.shift.intensive.template.dto - объекты, необходимые для взаимодействия с API
5. Пакет ru.cft.shift.intensive.template.exception - кастомные исключения
6. Пакет ru.cft.shift.intensive.template.repository - классы для взаимодействия с БД
7. Пакет ru.cft.shift.intensive.template.service - классы с бизнес-логикой (по сути связь API и БД)

## Общие требования к проекту

### Технические

1. Использование Java 17
2. Использование Spring Boot 3
3. Использование Spring Security. Данные пользователя (логин и пароль) хранятся в зашифрованном виде. Наличие прав/ролей
4. Использование Docker
5. Использование Apache Cassandra 4 (скрипты в каталоге scripts, название схемы - фамилия латинскими буквами)
6. Использование системы автоматической сборки Gradle
7. Использование фреймворка логирования (желательно Logback)
8. Наличие документации Swagger UI (OpenAPI)
9. Наличие коллекции запросов для Postman для проверки API
10. Наличие README, в котором описан проект, как его запустить, функциональность этого проекта и, если это необходимо,
    тестовые данные для запуска

### Проектные

1. Приложение запускается и реализован функционал задачи
2. Перед реализацией задачи сделано и согласовано API с куратором в текстовом виде
3. Перед реализацией задачи сделана и согласована схема БД в Apache Cassandra с куратором
4. Код задачи в репозитории в отдельной ветке, по возможности создан PR в мастер
5. Код чистый (нет мусорных/тестовых методов, закомментированного кода, используются читаемые и понятные имена
   переменных, соблюдается Java Naming Convention, есть декомпозиция, соблюдение принципа SOLID)
6. Аутентификация и авторизация пользователей
7. Валидация всех входных данных API (например, нельзя вставить строковое значение в числовое поле, проверка
   обязательности полей, проверка на количество символов и т.д;)

### Займы

Коммерческая организация занимается предоставлением займов физическим лицам в валюте РФ. Производственной необходимостью
является создание сервиса  (сайт / приложение) для более удобной работы пользователей. Сервис должен позволять:

1. оформить займ;
2. погасить займ;
3. просмотреть информацию по займам в личном кабинете.

При оформлении займа пользователь должен иметь возможность ввести сумму займа. В зависимости от суммы сервис
автоматически определяет процентную ставку и рассчитывает срок погашения займа. После оформления займа сумма
зачисляется
на личный счет.

Для погашения займа пользователь вводит сумму гашения. Сумма списывается с личного счета. При этом производиться
перерасчет суммы долга.

В личном кабинете пользователя должна быть возможность просмотреть все суммы долга по займам пользователя, количество
средств на личном счете пользователя. Личный счет можно пополнить, например, промокодами, можно вывести средства на
любой другой банковский счет.

## Задания личные

### Бронирование номеров в отеле

Необходимо написать систему в которой будут участвовать отели, постояльцы, коллекции номеров в каждом отеле, цены и
доступность номеров. Так же отели предлагают список достопримечательностей: парки, музеи, крупные магазины, памятники и
т.д.

#### Разбиение по ролям:

1. Администратор
2. Постоялец/гость

#### Требования к роли Администратор:

1. Добавление/редактирование отеля
2. Добавление/редактирование достопримечательностей рядом с отелем
3. Добавление/редактирование номеров в отелях
4. Добавление/редактирование принадлежностей в номерах
5. Получение номеров в отеле (отдельно свободных на заданные даты, отдельно занятых)
6. Получение броней по отелю на дату

#### Требования к роли Администратор (дополнительно):

1. Добавление фото к отелю (храним на файловой системе + метаданные в БД)
2. Добавление фото к номерам отелей (храним на файловой системе + метаданные в БД)

#### Требования к роли Постоялец/гость:

1. Поиск отелей поблизости от заданной достопримечательности
2. Получение информации о заданном отеле, например его название и местоположение
3. Получение достопримечательностей близи заданного отеля
4. Поиск свободных номеров в заданном диапазоне дат
5. Получение информации о номере (описание + цена + принадлежности)
6. Получение брони по номеру подтверждения
7. Получение всех броней по имени гостя
8. Просмотреть подробную информацию о госте

#### Требования к роли Постоялец/гость (дополнительно):

1. Отмена бронирования номера в отеле
